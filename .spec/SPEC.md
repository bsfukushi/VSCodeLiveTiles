# SPEC: CC 状態バッジ（v0.6 目標）

## 1. 背景・目的

複数の VSCode で Claude Code を並列運用しており、AskUserQuestion や承認ダイアログで
止まったウィンドウへ即座に飛ぶことが本ウィジェットの存在目的。
現状はライブサムネイルで「見れば分かる」が、**どのウィンドウが入力待ちかは目視で探す**必要がある。

CCPet（前身の常駐アプリ）が構築したフックインフラは今もグローバル
`~/.claude/settings.json` で稼働しており、CC の全ライフサイクルイベントが
`~/.ccpet/events.jsonl` に追記され続けている。これを読むだけでバッジが実現できる。

**決定事項（2026-07-07 ヒアリング）**
- 状態粒度: 4状態フル（質問待ち / 承認待ち / 完了 / 作業中）
- 見せ方: キャプション帯バッジ ＋ 待ちタイルは枠色グロー。タイル位置は固定（並べ替えしない）
- CCPet 本体は引退し LiveTiles に一本化。フック（append-event.mjs）はそのまま使い続ける

## 2. データソース

### events.jsonl（読み取り専用）

- パス: `%USERPROFILE%\.ccpet\events.jsonl`（`CCPET_EVENTS_FILE` 環境変数があればそちら）
- 形式: 1行 = 1 JSON レコード `{ts, type, sessionId, projectName, cwd, payload}`
  - `ts`: ms epoch / `sessionId`: CC セッションID / `cwd`: フック実行時の作業ディレクトリ
- 書き手: `S:/Desktop Appli/CCPet/src/hooks/append-event.mjs`（グローバル hooks から起動）
- **LiveTiles は絶対に書き込まない**。追記専用ファイルの tail リーダーに徹する

### 受信イベントと意味

| type | 意味 |
|---|---|
| `pre_tool_use` | AskUserQuestion 呼び出し＝**質問待ち**（AUQ 以外はフック側で除外済み） |
| `permission_request` | 承認ダイアログ表示＝**承認待ち**（自動承認では発火しない） |
| `notification` | 入力待ち通知（許可要求・アイドル60秒等）＝**承認待ち**扱い |
| `stop` | 応答完了＝**完了** |
| `user_prompt_submit` | ユーザーが入力＝**作業中**（待ち・完了の解除） |
| `post_tool_use` | ツール実行完了＝**作業中**（AUQ 回答後の解除もこれで拾える） |
| `session_start` | セッション開始＝状態なし（入力待ちアイドル） |
| `session_end` | セッション終了＝状態破棄 |

## 3. 状態機械

### セッション単位（sessionId ごと）

**「同一セッションの最新イベントが状態を上書きする」** だけの単純規則。

```
質問待ち   ← pre_tool_use
承認待ち   ← permission_request | notification
完了       ← stop
作業中     ← user_prompt_submit | post_tool_use
なし       ← session_start | session_end（end は破棄）
```

- 遷移の妥当性検証はしない（イベント欠落があっても次のイベントで自然回復する）
- **鮮度切れ**: 最終イベントから 24 時間超のセッションは破棄（クラッシュで
  session_end が来ないケースの掃除）

### ウィンドウ単位（表示状態）

1 ウィンドウに複数セッションが対応付いた場合は優先度で代表を選ぶ:

```
質問待ち ＞ 承認待ち ＞ 完了 ＞ 作業中 ＞ なし
```

## 4. cwd → VSCode ウィンドウの対応付け

- タイルのキャプション（サフィックス除去後）は VSCode 既定タイトル
  `ファイル名 - フォルダ名` の形式
- 照合規則: **除去後キャプションが `basename(cwd)` で終わる**（大文字小文字無視）
- 補助照合: `projectName` でも同様に試す（マルチルートワークスペース対策）
- マッチしない場合は**バッジなし**（誤表示より無表示を優先）
- 同一フォルダを複数ウィンドウで開いている場合は両方に同じバッジを出す（許容）

## 5. UI 仕様

### キャプション帯バッジ（全状態で表示）

キャプション帯の右端に「色ドット ＋ 状態ラベル」を表示:

| 状態 | 色 | ラベル |
|---|---|---|
| 質問待ち | 黄 `#E5C07B` | `❓ 質問 4:32`（経過時間付き） |
| 承認待ち | 橙 `#E06C75` 系オレンジ | `🔑 承認 0:15`（経過時間付き） |
| 完了 | 緑 `#98C379` | `✔ 完了` |
| 作業中 | 青 `#61AFEF` | `● 作業中`（控えめ） |
| なし | — | 非表示 |

- 経過時間は状態遷移時刻（イベント ts）からの `m:ss`。1 秒間隔の DispatcherTimer で更新
  （待ち状態のタイルが 1 つもないときはタイマー停止）
- 帯のフォントサイズ 24px に合わせてバッジも視認性優先

### 枠色グロー（待ち状態のみ）

- 質問待ち / 承認待ちのタイルは BorderBrush を状態色にし、ゆっくり明滅（約 1Hz、
  色の不透明度アニメーション）
- **BorderThickness は 2 固定を維持**（太さを変えるとサムネイル矩形がズレる。
  ThumbnailTile.cs:52 の既存制約）
- 強調の優先度: **未確認の待ちグロー ＞ アクティブ（VSCode ブルー）＞ ホバー ＞ 通常**
- 明滅するのは**未確認の待ち**だけ（§8「承認待ちバッジの解除タイミング」参照）
- タイルの位置・順序は変更しない（筋肉の記憶を壊さない）

## 6. 実装構成（新規ファイル）

```
Services/
├── CcEventsReader.cs   # events.jsonl の tail リーダー
│                       #   起動時: 末尾 512KB を遡って読み、状態を再構築
│                       #   稼働中: FileSystemWatcher + 保存オフセットから追記分のみ読む
│                       #   オフセット > ファイル長ならローテーションとみなし先頭から再読込
│                       #   壊れた行・パース失敗行はスキップ（フック側の設計前提と同じ）
└── CcStateTracker.cs   # sessionId → 状態の機械 ＋ cwd 照合 ＋ ウィンドウ単位の代表状態算出
                        #   状態変化時のみイベント発火（MainWindow がタイルに配布）
```

- ThumbnailTile に `SetCcState(state, since)` を追加。ApplyHighlight の優先度に待ちグローを組み込む
- 失敗時方針: events.jsonl が無い / 読めない場合は**バッジ機能を静かに無効化**
  （タイル・クリックの既存機能には一切影響させない）

## 7. スコープ外（今回やらない）

- append-event.mjs の LiveTiles リポジトリへの移設（CCPet パス参照のままで動くため。
  CCPet ディレクトリを整理する時に別タスクとして実施）
- VSCode 以外のアプリへのバッジ適用
- 通知音・トースト（既存のグローバル hooks が既に鳴らしている）
- タイルの並べ替え・フィルタ

## 8. 確定済み事項（旧・未決事項）

- 「完了」バッジの寿命: **次の user_prompt_submit まで残し続ける**（2026-07-07 承認）
- 「承認待ち」バッジの解除タイミング（2026-07-09 承認 / v0.6.2）
  - Claude Code には**承認した瞬間に発火するフックが存在しない**。ライフサイクルは
    `PreToolUse → PermissionRequest → PostToolUse` で `PreToolUse` は承認ダイアログの前に走る。
    承認の成立を示す最初のイベントはツール完了時の `post_tool_use` になる
  - 実測（events.jsonl 直近 721 件）: `permission_request` → `post_tool_use` の遅延は
    中央値 15.1 秒 / p90 71.3 秒 / 最長 369.1 秒。この遅延の正体は**ツールの実行時間**であり、
    イベントの取りこぼしではない（後続 121/124 件が `post_tool_use`）
  - 対処: 承認するにはそのウィンドウを前面に出すしかないため、**前面化を「確認した」とみなして
    待ちグロー（明滅）だけを止める**。バッジ本体は `post_tool_use` まで残す
    （実行中であることは経過時間で読める）
  - 非アクティブ化では確認済みフラグをリセットしない。承認直後に別ウィンドウへ移ると
    明滅がぶり返すため。次の待ち（`SinceTs` 更新）で自然に解除される
  - 前面化の検知は既存の `WindowTracker.ActiveWindowChanged` に相乗りする。
    タイルのクリックでも Alt+Tab でも同じ経路を通る
- 代表状態の優先度で **Done < Working** とする（2026-07-09 承認 / v0.6.3）
  - タイルとセッションは 1 対 1 ではない（照合は cwd の basename / projectName の suffix 一致）。
    同一プロジェクトを別ウィンドウで開くと 1 タイルに複数セッションがぶら下がる
  - 旧優先度は `Done`(2) > `Working`(1) だったため、**終わったセッションの「完了」が
    動いているセッションの「作業中」を上書きしていた**（2026-07-09 に PropBoard で実発生）
  - 完了は過去の結果、作業中は現在進行。片方が動いているならタイルは「作業中」を出す。
    単一セッションのタイルでは挙動が変わらないので「完了バッジは次の user_prompt_submit まで残す」
    という上記の確定事項とは衝突しない
  - 既知の残課題: 死んだセッションが `WaitingPermission` のまま残ると、そのタイルは
    `StaleAge`（24 時間）ずっと承認待ちに固定される。塞ぐなら `Resolve` 側で
    「一定時間更新のないセッションを代表選出から外す」
- `stop` は「完了」とは限らない（2026-07-09 承認 / v0.6.4）
  - `stop` はメインエージェントのターン終了にすぎず、**サブエージェント（`type: subagent`）や
    `run_in_background` の Bash（`type: shell`）は走り続ける**
  - `stop` の `payload.background_tasks` に、その時点で生きているタスクが入る。
    終わったタスクはリストから消えるため `status` は `running` しか現れない
  - よって `stop` かつ running な背景タスクあり → `Working`、なし → `Done` とする
  - 実測（2026-07-09 PropBoard）: 背景 4 件 → 2 件 → 0 件と減り、最後の `stop` で `Done` に落ちる。
    サブエージェント完了を示すイベント（`SubagentStop`）をフックに足さなくても自然に閉じる

## 9. セッション時計（v0.7.0 / 承認 2026-07-09）

キャプション帯に `開始時刻 ▸ 経過`（例 `11:37 ▸ 5:23`）を常時表示する。
帯は「キャプション（可変幅）／時計／状態バッジ」の 3 列。
時計とバッジは同じ 20px。時計は灰色にして主張を抑える（v0.7.1 で 16px から変更）。
アクティブ帯（VSCode ブルー）の上では時計も白にする。

### 開始時刻の取得

- 出典は `session_start` の `ts`
- `resume` で `session_start` が再発火する（実測 27 セッション中 4 件）。
  **最初に観測した方を採用**し、後の `resume` では上書きしない
- `compact` による再発火は観測されていない
- **状態の再構築（末尾 512KB）では開始時刻に届かない**（実測: 生存セッション 6 件中 5 件が範囲外）。
  起動時に events.jsonl 全体を 1 度だけ走査し、`"session_start"` を含む行だけ JSON パースする
  （実測 24.3MB / 2929 行 / 38 件 → 64ms）
- `transcript_path` のファイル作成時刻は使えない。`session_start` から最大 3.5 時間ずれ、
  ファイル自体が存在しないケースも 27 件中 9 件あった
- ローテーションで `session_start` が流れたセッションは時計を出さない（推測しない）

### 更新

- 既存の 1 秒 DispatcherTimer に相乗り。稼働条件を「待ちタイルあり」から
  「待ちタイルあり **または** 時計を持つタイルあり」に広げる
- 経過は分単位でしか変わらないため、文字列が同じなら `TextBlock.Text` に代入しない（再描画抑止）
