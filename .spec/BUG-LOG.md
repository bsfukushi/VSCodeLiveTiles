## 2026-07-10 タイルクリックで約10秒フリーズしてから切り替わる（v0.7.3）

- **症状**: タイルクリック→画面切り替え時に、ウィジェットごと約10秒フリーズすることが頻発
- **原因**: `MoveMaximizeAndFocus` が UI スレッド上で `ShowWindow` / `SetWindowPos` /
  `AttachThreadInput` を同期実行。これらは相手ウィンドウのウィンドウプロシージャの応答を
  待つため、切り替え先の VSCode（Electron）の UI スレッドが詰まっていると呼び出し元まで
  巻き添えブロックされる。CC フル稼働中のセッションで VSCode が数秒詰まるのが引き金
- **修正**: `ShowWindow` → `ShowWindowAsync`、`SetWindowPos` に `SWP_ASYNCWINDOWPOS`
  （相手のキューに積むだけ。キューは順番どおり処理されるので 復元→移動→最大化 の順序は保たれる）。
  `AttachThreadInput` は廃止 — クリック直後は自プロセスがフォアグラウンドなので
  素の `SetForegroundWindow` で通る
- **学び**: 別プロセスのウィンドウを操作する Win32 API は「相手を待つ同期版」と
  「積むだけの非同期版」がある。UI スレッドから呼ぶなら非同期版が原則。
  `AttachThreadInput` は相手が詰まると自分の入力処理まで止まる「人質 API」

## 2026-07-10 タイル切替だけ画面がガチャっとずれながら再描画される（v0.7.4）

- **症状**: タスクバーからの切り替えはきれいなのに、タイルからだと
  「一瞬縮む→動く→最大化し直す」ような再描画が見える
- **原因**: ウィンドウの現在地を見ずに、クリックのたびに無条件で
  復元→移動→最大化 の3段階を実行していた。すでに対象モニターで最大化済み
  （普段の切り替えのほぼ全ケース）でも毎回位置・サイズを触っていた
- **修正**: `MonitorFromWindow` / `MonitorFromPoint` で現在地を判定し、
  対象モニターで最大化済みなら `SetForegroundWindow` だけ／対象モニター上なら最大化のみ／
  別モニターのときだけ従来の3段階。最小化中は現在座標が画面外（-32000）なので
  `GetWindowPlacement` の通常時矩形（rcNormalPosition）でモニターを判定
- **学び**: 「A 経由だときれい、B 経由だとガチャつく」という比較つきの体感報告は
  ほぼ二分探索。A（タスクバー＝前面化だけ）との差分を見るだけで原因が確定した

## 2026-07-10 クリック直後にウィジェット自身がホワイトアウトして「応答なし」になる（v0.8.1）

- **症状**: タイルをクリックした直後、ウィジェットが薄い白でホワイトアウトし、中央に
  Windows の「プログラムを終了しますか」ダイアログが出る。数秒待つと復帰して普通に使える（2回発生）
- **切り分け**: 「薄い白＋終了しますか」は Windows が **5秒以上メッセージを処理しないウィンドウ**に
  被せるゴーストウィンドウ。ただし白くなったのがウィジェット側か VSCode 側かで原因が正反対になるため、
  そこだけユーザーに確認した（→「ウィジェット側」「タイルをクリックした直後」）
- **原因**: v0.7.3 でクリック経路の同期 Win32 呼び出しを非同期化したとき、`SetForegroundWindow` だけが
  同期のまま残っていた。これは相手スレッドのアクティベーション処理を待つため、CC で重い処理中の
  VSCode が応答しないと**ハングアプリ判定タイムアウト（5秒）**まで戻ってこない。
  10秒→5秒に減っていたのはそのため
- **修正**: `MainWindow.OnTileClicked` から `MoveMaximizeAndFocus` を `Task.Run` で背景スレッドへ。
  フォアグラウンド化の権利はプロセス単位なので、別スレッドから呼んでも `SetForegroundWindow` は通る。
  あわせて所要時間を計測し、200ms 以上なら「相手が応答していない」事実をログに残す
- **学び**: 同種の修正は「全部潰したか」を必ず数え上げる。3つ非同期化して満足し、4つ目を見落とした。
  v0.7.3 のコメントに「呼び出し元をブロックしない」と書いてあったのに、その直下に同期呼び出しが残っていた

## 2026-07-10 events.jsonl の読み取りが UI スレッドを最大 9 秒止める（v0.8.3）

- **症状**: 上記 v0.8.1 を入れた直後、ウォッチドッグが 8.4 秒の UI スレッド停止を記録。
  `SetForegroundWindow` の警告は出ておらず、別の犯人がいると判明。本番実測で `Drain 9155ms`、
  Send / Input の両方が同時に停止（＝Dispatcher キューの混雑ではなく、真のブロック）
- **原因**: `CcEventsReader.Drain()` が UI スレッド上でファイルを open / read / parse していた。
  `~/.ccpet/events.jsonl` は際限なく育ち、この時点で 42MB。**追記直後の open だけで 140ms** かかる
  （実測: 同サイズのダミーで 138〜153ms、2回目のキャッシュ済み open は 0.1ms、小さいファイルなら 7ms
  → スキャナがファイル全体を舐めている）。ディスクとスキャナが混雑すると数秒に膨らむ。
  FileSystemWatcher と 1 秒ポーリングの両方が、これを UI スレッドで踏んでいた
- **修正**: ファイル I/O とパースを全てバックグラウンドへ。`DispatcherTimer` → `System.Threading.Timer`、
  watcher 通知もワーカーへ投げる。読み取りは `Monitor.TryEnter` で 1 本に直列化し、取れなければ捨てる
  （取りこぼしは次のポーリングが拾う）。UI スレッドへ渡すのはパース済みレコードだけ
- **検証**: 40 万行を一気に追記して Drain を強制的に重くする A/B。
  修正前 = Drain 1340ms（UI スレッド）→ Send 1313ms / Input 1360ms 停止。
  修正後 = Drain 1577ms（背景スレッド）→ UI スレッドの停止 0 件
- **外した仮説**: ①`Process.GetProcessById`（ウィンドウ列挙）→ 8 PID 合計 0.29ms で無罪。
  PowerShell で測ると 47ms に見えたが、それは PowerShell のオブジェクト生成コストだった。
  ②巨大な JSON 行 → 最大 635KB、42MB 全行パースでも 300〜400ms で無罪。
  ③相手ウィンドウのハング → 20 秒メッセージを処理しない偽 `Code.exe` をぶつけたが、
  ウィンドウ列挙も DWM も一切詰まらなかった
- **学び**: **直す前に観測手段を入れる**。ログと UI スレッド ウォッチドッグを先に入れたおかげで、
  1つ目の修正が不十分だと 37 秒で分かった。ウォッチドッグは Send / Input の 2 優先度で ping を打つと
  「ネイティブ呼び出しでブロック」と「キューの混雑」を区別できる（直す場所が正反対）。
  そして**測る道具を疑う** — 仮説より先にベンチを検算する

## 2026-07-10 バッジ更新が最大 20 秒遅れる — 背景スレッドに逃がした先で open が育ち続けていた（v0.8.4）

- **症状**: v0.8.3 リリース後の常用ログに `events.jsonl の読み取り 9556ms / 19854ms（背景スレッド）`
  の警告が並んでいた。内訳はほぼ 100% open。UI は固まらない（それは v0.8.3 で解決済み）が、
  読み取りが終わるまでバッジが更新されないので、CC の状態表示が最大 20 秒古いまま。
  ユーザーからの報告はなく、次セッション冒頭のログ確認で発見
- **原因**: 追記直後の open はスキャナがファイル全体を舐める（v0.8.3 調査で判明済み・サイズ比例）。
  `Drain()` は毎回 open→読む→閉じる を繰り返していたため、ファイルが 42→46.5MB と育つにつれ
  毎回の open コストも 140ms→20 秒へ悪化。「UI スレッドから外す」は完了していたが、
  「open を繰り返す」構造はそのまま持ち越していた
- **修正**: ファイルハンドルを開きっぱなしに（open は起動時の 1 回だけ）。
  ローテーションは同一ハンドルの長さの縮みで検知でき（truncate はハンドル越しに見える）、
  差し替え・削除・リネームは FileSystemWatcher の Created/Deleted/Renamed でハンドルを開き直す。
  読み取り例外時はハンドルを捨てて次回開き直す
- **検証**: 常用 3 分でイベント 1.2MB 流入、読み取り警告 0 件（修正前は同条件でほぼ毎回警告）
- **学び**: ①「スレッドを移す」と「コストを消す」は別の修正。逃がした先でコストは育ち続ける。
  遅い処理を背景に移したら、その処理自体の遅さも計測が残るようにしておく（今回それで救われた）。
  ②open が高いなら開き直さないのが最速。追記追跡の読み取りはハンドル保持が原則で、
  ローテーション検知は「長さの縮み＋ファイル名イベント」で足りる
